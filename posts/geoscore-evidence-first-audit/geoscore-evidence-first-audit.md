# 我为什么又做了一个 SEO / GEO 审计工具

以前我用过不少一键审计工具。它们的共同问题不是“检查项少”，而是太容易把不确定的信息伪装成结论：个人博客会被当成 SaaS，文章站会被推荐价格表或 `Service` schema；外部接口超时后，页面仍然显示 0 分和成功图标；所谓“AI 可见性”也常常被写得像已经监测到真实引用。

我想要的不是一张漂亮但无法复核的分数卡，而是一份能回答下面几个问题的报告：

- 这个站到底被识别成什么类型，依据是什么？
- 哪一个页面、哪一段结构或哪一个响应导致了失败？
- 这项检查是否真的适用于这个站？
- 外部数据缺失时，分数为什么没有被误扣？
- 修改后应当怎样复验？

因此做了 [GeoScore](https://github.com/Amiyadesi/geoscore)。它是 MIT 开源项目，在线入口在 [geo.sayori.org](https://geo.sayori.org)。项目 README 也明确认可并链接了 [LINUX DO](https://linux.do/) 社区。

# GeoScore 实际审什么

GeoScore 默认不是只抓首页。它会建立站点画像，并在站点模式下最多抽取五个有代表性的 HTML 页面：首页、可发现的 About 页，以及从 sitemap 或内部链接中按路径类型确定性选择的页面。单 URL 模式则审查指定页面，并在必要时读取首页作为上下文。

站点画像优先使用 JSON-LD、canonical、标题、导航和页面结构。正文里出现“AI”“Cloudflare”“服务”这样的字，不应该把带有 `Blog + Person` schema 的个人博客强行判成公司产品站。报告会保留类型、实体、语言、根域名、置信度和原始证据，用户也可以只对本次审计提交类型提示，不会把匿名纠正写成全局覆盖。

审计结果把每一项事实状态分成：

| 状态 | 含义 |
| --- | --- |
| `pass` | 已在指定页面或响应中验证通过 |
| `fail` | 有明确证据表明需要修复 |
| `not_applicable` | 对当前站点类型或页面类型不适用 |
| `unknown` / `error` | 证据源不足或外部服务失败，不能当成失败 |

SEO 侧会覆盖可索引性、robots 与 sitemap、canonical、metadata、语义结构、schema 适配性、跨页面一致性与性能证据。GEO Readiness 侧关注实体是否清晰、作者和来源信号、正文是否容易提取、直接回答结构、声明与来源关联、统计引用和内容时效性。

评分只从“已知且适用”的 `pass` / `fail` 检查计算；`unknown`、`error` 和不适用项不会偷偷扣分。报告同时给出 coverage 与 confidence：证据不足时，总分是 `null`，显示“证据不足”，而不是用 0 分假装得到负面结论。

# 不把预测当成真实 AI 引用

GEO 很容易被营销话术带偏，所以 GeoScore 把两类输出拆开：

1. **事实 GEO Readiness**：来自页面和可验证外部证据，可进入主分。
2. **Predicted visibility**：基于内容主题与查询机会的模拟，明确标为 `Predicted`，权重固定为 0，不影响 SEO、GEO 或 overall 主分。

它不会声称自己已经监测到了 ChatGPT、Perplexity 或 Google AI Overview 的真实引用结果。搜索网关可以帮助发现主题簇和内容机会，但搜索结果不是 AI 引用监控的替代品。

# 报告不应该只告诉你“去优化”

每条建议都必须来自适用的 `fail`，并且包含：在哪个页面发现了什么、原始证据、为什么失败、怎样修改、怎样复验。已经存在的 schema 不会再次被推荐；个人博客和 editorial 站点也不应该收到“加套餐、价格或 Service schema”的模板化建议。

审计完成后可以直接下载完整 Markdown 修复报告。它会收集站点画像、抽样页面、评分及封顶原因、全部适用失败、未知或外部错误、可执行的修复与复验步骤，并给出一个不虚构业务事实的开发 AI handoff prompt。它不自动修改、提交或发布被审计的网站。

# Lighthouse 和外部证据也要诚实

PageSpeed Insights 的 mobile 和 desktop 被分别建模：一端成功时报告显示 `partial`，保留真实分数和另一端错误；两端都失败时，API 返回结构化错误和是否可重试的标记，不再返回 `ok: true` 或伪造 0 分。可选的 Common Crawl、W3C 校验、CrUX、Browser Run 等证据源同样如此：失败会被展示为证据不足或错误，而不是变成对网站的惩罚。

# 开源和后续方向

代码、许可证和上游署名都在 GitHub：<https://github.com/Amiyadesi/geoscore>。当前目标是把审计做得更可复核、更适合不同类型的网站，而不是堆更多“看起来很智能”的固定建议。

如果你有个人博客、文档站、SaaS、电商、本地业务或内容媒体站点，也欢迎拿自己的 URL 试一试。如果类型识别或建议不合理，把报告里的证据一起带回来讨论会很有帮助：我更希望修正可验证的规则，而不是用一个新模板覆盖所有站点。
